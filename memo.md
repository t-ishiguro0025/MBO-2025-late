- この画面の目的は何か

## 2. 実装対象と実装順

1. まず設計に基づいて基礎を実装する（理由：ディレクトリの構成を確定させて、構成を理解しやすくするため）
2. 次にスキーマを実装する（理由：後々のコンポーネント実装時にデータを受け取る前にスキーマによる検証を行いたいから）
3. 次にログイン画面に必要なコンポーネントを実装する（理由：定義したスキーマを使用して、ログイン画面に必要なコンポーネントを作成する必要があるから）
4. 次にログイン画面コンポーネント（Presentation）を実装する（理由：機能がない状態で、正しいログイン画面が作成できるか確認する必要があるから）
5. 次にAPIを実装する（理由：ログインボタンを押した時の機能をContainerコンポーネント作成前に実装する必要があるから）
6. 次にログイン画面コンポーネント（Container）を実装する（理由：ボタンを押下した時の送信処理や、メール・パスワードの検証を行なった画面コンポーネントを最終的に実装してログイン画面を完成させる必要があるから）

## 5. 実装後の振り返り

- 設計との差分
  AIが生成したコンポーネント構成案では以下のような階層になっていた：
  
  LoginPage
  └── LoginForm
      ├── EmailField
      ├── PasswordField
      ├── ApiErrorAlert
      └── SubmitButton

  しかし実際の実装では、UIと機能（状態管理・送信処理）が一体になってしまい、
  役割の分離ができないと判断した。
  そのため Presentation / Container の構成に変更した。

- 実装判断の理由
  - 階層が深くなりすぎるとコードの見通しが悪くなる
  - UIと機能が一体だとメンテナンスやレビューがしにくい
  - PresentationはUIの描画のみ、ContainerはAPIや状態管理のみに責務を分けることで、
    それぞれ独立して変更・テストできる構成にした

- 次回への改善点
  - 初期設定・環境構築周りは実装開始前に「何を使うか・何のために追加するか」を
    決めておく。途中で変更するとそれに応じた対応が多く発生するため
  - APIやモック周りの設定・扱い方は事前に調べておく
  - 型定義は種類が多いため、使えるパターンと使い方をある程度インプットしてから
    実装に入る

## 7. 詰まったポイント

- zod min　（必須項目としての役割） と zod email・zod passwordのエラー文言が両立してしまう

参考: https://blog.asobou.co.jp/web/zod-01

に詰まった。
公式ドキュメント見ても解決しなさそうだった。

解決策としては、

```ts
.min(1, { message: "メールアドレスを入力してください" })
.refine((v) => z.string().email().or(z.literal("")).safeParse(v).success,
{
    message: "正しいメールアドレスを入力してください",
}),
```

「””」も success として検証を通す。
